# CLAUDE.md — md-fire

Native macOS Markdown editor fusing **Typora** (live WYSIWYG) with **iA Writer** (focus, typography,
restraint). Swift + SwiftUI + AppKit/TextKit 2. The user wants it premium ("ахуенный") and built
ambitiously. Communicate with the user in **Russian**.

## How to build / run (I cannot use the Xcode GUI)

The `.xcodeproj` is **generated from `project.yml` via XcodeGen** (the project is git-ignored).

```sh
xcodegen generate                                                   # after editing project.yml or adding files
xcodebuild -project md-fire.xcodeproj -scheme md-fire -configuration Debug -destination 'platform=macOS' build
open ~/Library/Developer/Xcode/DerivedData/md-fire-*/Build/Products/Debug/md-fire.app   # run it
swift test                                                          # in Packages/MarkdownCore (fast, no app host)
```

`make project|build|run|open` also work. After adding/removing source files, **always `xcodegen generate`**
before building. To relaunch: `killall md-fire; open <app>`.

**Verification loop:** I can't screenshot (harness lacks Screen Recording). The **user pastes
screenshot file paths and I `Read` them**. After each build, relaunch and ask the user to look/test.
SourceKit "No such module" diagnostics in-editor are noise (SPM build context); trust `xcodebuild`.

## Architecture

- **`Packages/MarkdownCore`** (local SwiftPM, pure logic, unit-tested): `TreeSitterParser` (split
  block+inline grammars → `SyntaxNode` with content vs marker ranges), `RangeMapping`, `FocusScope`/
  `FocusRange`, `MarkupCollapse`. App depends on it.
- **App `Sources/MdFire/`**: `App/` (MdFireApp scene + menu commands), `Document/` (MarkdownDocument
  @Observable open/save, WorkspaceModel sidebar tree), `Engine/TextKitEngine/` (TextKitEditor =
  NSViewRepresentable over STTextView; Styler; StylePolicy; FillingTextView; EditorController),
  `Theme/`, `UI/` (RootView = NavigationSplitView; SidebarView; FormatBar), `Export/`, `Telegram/`.
- The editor is **STTextView** (`krzyzanowskim/STTextView` 2.3.10) in TextKit 2. The Coordinator in
  `TextKitEditor.swift` owns the parse + styling pipeline.

## CRITICAL gotchas (hard-won — don't relearn)

- **SwiftTreeSitter parses UTF-16LE**, so tree-sitter byte offsets are UTF-16 *bytes*; **÷2 = NSRange
  code units**. No UTF-8 conversion needed (NSTextStorage is UTF-16).
- **WYSIWYG marker hiding**: content-storage display-substitution BREAKS STTextView (assumes display
  length == storage length → blank). Working approach = **attribute-only**: hidden markers rendered
  at **~0.01pt transparent** font (collapse, gap-free, storage intact). Modes differ only in marker
  attributes. Caret-reveal restores full-size dim markers for the element under the caret.
- STTextView API: `text`, `addAttributes(_:range:)`, `textDelegate` (NOT delegate),
  `isHorizontallyResizable` (NOT widthTracksTextView), `textSelection` (NSRange, settable),
  `insertText(_:replacementRange:)`, `textLayoutManager`/`textContentManager` exposed. Subclassing:
  `scrollableTextView()` uses `Self()` so `FillingTextView` is picked up.
- **PDF export**: `NSPrintOperation.run()` with WKWebView **deadlocks/hangs** → use async
  `webView.createPDF`. **Telegram paste**: Telegram for macOS **ignores rich-text/RTF** → copy
  **plain Telegram-markdown** (`**bold**`, `__italic__`). Both reveal output in Finder.
- **Status bar overflow throws an AutoLayout NSException on launch** — keep it light (toggles live in
  one "view options" menu, not inline buttons). Re-triggered by **`.fixedSize()` on a wide status
  Text**: it forces the HStack wider than a narrow window's first layout pass. Use low
  `.layoutPriority(-1)` + `.truncationMode(.tail)` for stats text so it yields instead of overflowing.
- Per-keystroke heavy work (NLTagger POS, bionic) is **computed on text/toggle change and cached**,
  then re-applied cheaply on every restyle (restyle runs on each caret move).
